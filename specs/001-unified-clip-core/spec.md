# Feature Specification: 统一剪藏核心 (Unified Clip Core)

**Feature Branch**: `001-unified-clip-core`

**Created**: 2026-07-04

**Status**: Draft

**Input**: User description: "结合现有项目功能,确定并规范化统一剪藏核心功能,覆盖网页/视频/图片/文档的端到端剪藏流程"

## Clarifications

### Session 2026-07-04

- Q: 去重身份语义——当用户再次剪藏一个之前剪藏过、但内容已变化的网页时如何处理? → A: 视为重复,提示已存在并询问是否覆盖(刷新归档)
- Q: 归档目录命名冲突(同标题同时间戳)如何解决? → A: 自动追加数字后缀(如 `时间戳_标题-1`)
- Q: 索引文件损坏或被外部修改时如何恢复? → A: 备份损坏索引后,重建空索引并继续剪藏,日志记录恢复事件,同时向用户提示
- Q: 批量剪藏中单条失败如何处理? → A: 自动重试 1 次,仍失败则跳过并继续后续条目,在回执中标记失败原因
- Q: 输入文件大小与视频时长上限阈值? → A: 单文件 20MB、视频时长 15 分钟

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 网页剪藏 (Priority: P1)

用户在命令行输入一个网页网址,系统抓取该网页正文转为 Markdown、下载内嵌图片、生成整页长截图,并按内容领域自动归类存档,同时把原始来源与抓取时间写入索引以便回溯。

**Why this priority**: 网页是最高频、最易失效的剪藏对象(原页可能随时下线),端到端归档是项目存在的根基,因此列为最高优先级。

**Independent Test**: 给定一个可访问的网页网址,执行剪藏后,可在归档目录中找到正文 Markdown、下载的图片与整页截图,且索引文件包含该条记录。

**Acceptance Scenarios**:

1. **Given** 一个公开可访问的网页网址, **When** 用户对该网址发起剪藏, **Then** 系统在对应领域目录下生成正文 Markdown、下载内嵌图片、生成整页截图,并在索引文件中追加一条含原始网址与抓取时间的记录
2. **Given** 一个需要登录或已下线的网页网址, **When** 用户发起剪藏, **Then** 系统不崩溃,向用户输出明确的失败原因,且不产生半成品归档目录
3. **Given** 一个曾经剪藏过的网页网址, **When** 用户再次发起剪藏, **Then** 系统识别为重复并向用户提示已存在的归档位置,询问是否覆盖或跳过

---

### User Story 2 - 视频剪藏 (Priority: P1)

用户输入一个视频网址,系统获取视频信息、字幕与封面;当无官方字幕时,按分级回退策略自动进行语音转写,最终把视频信息、字幕文本与封面统一归档。

**Why this priority**: 视频内容的字幕是稀缺且易失的资源,自动转写回退链是项目差异化能力的核心,与网页剪藏并列为 P1。

**Independent Test**: 给定一个视频网址,执行剪藏后,可在归档目录中找到视频信息文件、字幕文本与封面图片;若该视频无官方字幕,归档目录中仍存在由语音转写生成的字幕文本。

**Acceptance Scenarios**:

1. **Given** 一个含官方字幕的视频网址, **When** 用户发起剪藏, **Then** 系统获取视频信息、官方字幕与封面并归档,不触发语音转写
2. **Given** 一个无官方字幕的视频网址, **When** 用户发起剪藏, **Then** 系统按分级回退策略依次尝试可用的语音转写后端,成功后将转写文本作为字幕归档,并在日志中记录实际使用的后端
3. **Given** 所有语音转写后端均失败的视频, **When** 用户发起剪藏, **Then** 系统仍归档视频信息与封面,在归档目录中标注字幕缺失,并向用户说明失败原因
4. **Given** 一个曾经剪藏过的视频网址, **When** 用户再次发起剪藏, **Then** 系统识别为重复并提示已存在的归档位置

---

### User Story 3 - 图片剪藏 (Priority: P2)

用户输入一张或多张本地图片路径(或图片网址),系统保存原图,并在输入多张时支持合并为一次归档;归档同时写入索引并按领域分类。

**Why this priority**: 图片剪藏频率低于网页与视频,但作为信息载体的价值同样重要,列为 P2。

**Independent Test**: 给定一张或多张图片,执行剪藏后,可在归档目录中找到原图存档,且索引文件含该条记录;多张图片输入时归档于同一目录。

**Acceptance Scenarios**:

1. **Given** 一张本地图片路径, **When** 用户发起剪藏, **Then** 系统保存原图至领域目录,并在索引中追加记录
2. **Given** 多张本地图片路径, **When** 用户发起剪藏, **Then** 系统将多张图片合并归档于同一目录,并在索引中记录图片数量
3. **Given** 一张与既有归档内容哈希相同的图片, **When** 用户发起剪藏, **Then** 系统识别为重复并提示已存在的归档位置

---

### User Story 4 - 文档剪藏 (Priority: P2)

用户输入一个 PDF 或 Word 文档文件路径,系统抽取文档正文、归档原文件,并按领域分类写入索引。

**Why this priority**: 文档剪藏面向需要长期留存正式资料的用户,价值稳定但频率低于网页与视频,列为 P2。

**Independent Test**: 给定一个 PDF 或 Word 文档,执行剪藏后,可在归档目录中找到抽取的正文与原文件存档,且索引文件含该条记录。

**Acceptance Scenarios**:

1. **Given** 一个可解析的 PDF 文档, **When** 用户发起剪藏, **Then** 系统抽取正文、归档原文件,并在索引中追加含来源路径与时间戳的记录
2. **Given** 一个受密码保护或损坏的文档, **When** 用户发起剪藏, **Then** 系统不崩溃,向用户说明无法解析的原因,且不产生半成品归档
3. **Given** 一个与既有归档内容哈希相同的文档, **When** 用户发起剪藏, **Then** 系统识别为重复并提示已存在的归档位置

---

### User Story 5 - 批量剪藏与去重回执 (Priority: P3)

用户一次性传入多个网址或多个文件路径,系统依次执行剪藏、对每条输入做去重检查,完成后输出一份汇总回执,列出成功、失败与重复条目。

**Why this priority**: 批处理是高频用户的效率倍增器,但依赖单条剪藏流程已稳定,因此列为 P3,在单条流程就绪后交付。

**Independent Test**: 给定一组混合类型的输入(网址与文件路径),执行批处理后,用户获得一份汇总回执,回执中每条输入对应明确状态(成功/失败/重复),且成功条目的归档目录与索引记录均存在。

**Acceptance Scenarios**:

1. **Given** 一组含多个网址与多个文件路径的输入, **When** 用户发起批量剪藏, **Then** 系统依次处理每条输入,完成后输出汇总回执,回执逐条列出成功、失败与重复状态
2. **Given** 批量输入中含与既有归档重复的条目, **When** 系统处理到该条目, **Then** 系统跳过实际抓取,在回执中标记为重复并附已存在的归档位置
3. **Given** 批量输入中某条目失败, **When** 系统完成整批处理, **Then** 失败条目自动重试 1 次,仍失败则跳过且不影响其他条目处理,回执中对该条目附失败原因并标注已重试

---

### Edge Cases

- 当输入网址需要登录、被反爬限制或返回非预期内容时,系统识别失败并输出明确原因,不产生半成品归档目录
- 当再次剪藏的网址内容已更新(网址相同但内容哈希不同)时,系统提示用户选择刷新归档或跳过,不直接覆盖
- 当外部抓取或语音转写服务出现限流或超时时,系统降级到下一个可用后端或生成失败占位,不在单条失败上无限重试
- 当输入文件超过 20MB 或视频时长超过 15 分钟时,系统提前拦截并给出明确提示,不进入抓取或转写流程
- 当归档目录名因标题清洗后产生冲突(同标题同时间戳)时,系统自动追加数字后缀(如 `-1`)以避免覆盖,无需用户介入
- 当输入为无法识别类型(既非已知网址、也非支持文件)时,系统拒绝剪藏并向用户输出明确提示,不进入抓取或解析流程
- 当索引文件损坏或被外部修改时,系统备份损坏文件后重建空索引并继续剪藏,在日志中记录恢复事件,同时向用户提示索引已重建及备份位置

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 接受单个网址或文件路径作为输入,并自动识别其剪藏类型(网页/视频/图片/文档)。无法识别类型(既非已知网址模式、也非支持文件后缀)时 MUST 拒绝剪藏并向用户输出明确提示,不得进入抓取或解析流程
- **FR-002**: 网页剪藏 MUST 抓取正文并转换为 Markdown,同时下载内嵌图片
- **FR-003**: 网页剪藏 MUST 默认生成整页长截图作为归档的一部分,用户 MUST 能通过配置关闭截图行为;关闭截图时不得影响正文 Markdown 与内嵌图片的归档
- **FR-004**: 视频剪藏 MUST 获取视频信息、字幕与封面;无官方字幕时 MUST 按分级回退策略自动进行语音转写。v1 以现有已支持的平台(以 B站为主)为范围,架构 MUST 为其他平台预留可扩展的接入点,但不在本规格内实现多平台
- **FR-005**: 语音转写分级回退链 MUST 依次尝试可用后端,每个后端的成功或失败 MUST 记录于日志
- **FR-006**: 所有语音转写后端均失败时,系统 MUST 仍归档视频信息与封面,并在归档中明确标注字幕缺失
- **FR-007**: 图片剪藏 MUST 保存原图,并在多张图片输入时合并归档于同一目录
- **FR-008**: 文档剪藏 MUST 抽取 PDF 与 Word 文档的正文,并归档原文件
- **FR-009**: 每条剪藏记录 MUST 按预定义的内容领域自动分类,归档至对应领域目录
- **FR-009a**: 归档目录命名以"时间戳_标题"为基本规则;当清洗后的目录名与既有目录冲突时,系统 MUST 自动追加数字后缀(如 `-1`、`-2`)以避免覆盖,无需用户介入
- **FR-010**: 系统 MUST 对每条输入进行去重检查,以网址作为去重主身份、以内容哈希作为辅证;命中时提示已存在归档位置并询问用户是否覆盖(刷新归档)。若内容哈希相同则视为完全重复,若网址相同但内容哈希不同则视为"内容已更新"并提示用户选择刷新或跳过
- **FR-011**: 批量剪藏 MUST 支持一次传入多个网址或文件路径,并在完成后输出逐条状态的汇总回执
- **FR-011a**: 批量剪藏中单条输入失败时,系统 MUST 自动重试 1 次;仍失败则跳过该条并继续处理后续条目,不得因单条失败终止整批;重试与最终失败均 MUST 在汇总回执中记录,回执中对该条目标注失败原因
- **FR-012**: 每条成功剪藏 MUST 在索引文件中追加一条记录,含原始来源、抓取时间戳、抓取后端与归档位置
- **FR-013**: 外部依赖失败时系统 MUST 优雅降级,不得直接崩溃,且不产生半成品归档目录
- **FR-013a**: 当索引文件损坏或无法解析时,系统 MUST 将损坏文件备份(保留原内容供事后分析),重建空索引并继续完成当前剪藏操作,在日志中记录恢复事件,并向用户输出明确提示说明索引已重建及备份位置
- **FR-013b**: 系统 MUST 对输入文件进行前置大小校验:单文件超过 20MB 时 MUST 拒绝剪藏并向用户输出明确提示;视频时长超过 15 分钟时 MUST 拒绝语音转写并向用户输出明确提示,不得进入抓取或转写流程
- **FR-014**: 系统 MUST 提供基于关键词的已剪藏内容检索能力,以便用户回溯已归档内容
- **FR-015**: 系统 MUST 提供统计信息查看能力,包含已剪藏条目总数与各领域分布

### Key Entities

- **ClipRecord(剪藏记录)**: 一次剪藏操作的完整记录,包含输入来源(网址或文件路径)、剪藏类型、所属领域、抓取时间戳、抓取后端、归档位置、去重状态
- **ClippedArtifact(剪藏产物)**: 一次剪藏产生的具体文件,如正文 Markdown、内嵌图片、整页截图、视频信息文件、字幕文本、封面图片、原文件存档;一个 ClipRecord 可对应多个 ClippedArtifact
- **IndexEntry(索引条目)**: 索引文件中的一条记录,指向一个 ClipRecord,含原始来源、时间戳、领域与归档路径,用于检索与回溯
- **DedupRecord(去重记录)**: 用于判重的记录,以网址哈希作为主身份、内容哈希作为辅证,支撑 FR-010 的查重与"内容已更新"判定能力
- **BatchReceipt(批量回执)**: 批量剪藏的汇总结果,含每条输入的状态(成功/失败/重复)、失败原因与归档位置

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户对单个常规网页发起剪藏,从执行到获得完整归档(含正文、图片、截图)耗时不超过 30 秒
- **SC-002**: 视频剪藏的字幕获取成功率(含官方字幕与语音转写回退)达到 95% 以上
- **SC-003**: 自动分类与用户预期领域的一致率达到 80% 以上
- **SC-004**: 对已剪藏过的网址或内容,系统识别为重复的命中率达到 90% 以上
- **SC-005**: 批量剪藏 10 条混合输入时,系统在 5 分钟内完成全部处理并输出汇总回执
- **SC-006**: 外部服务失败时,系统 100% 优雅降级,不丢失已抓取内容,不产生半成品归档
- **SC-007**: 用户在归档完成后,能通过单条关键词在 2 秒内检索到对应的已剪藏内容

## Assumptions

- 用户拥有稳定的网络连接,且可访问目标网页、视频平台与外部抓取服务
- 用户已按项目说明完成配置文件填写(含必要的 API Key),并已安装所需外部依赖
- 本规格范围限于单机本地存储,不含云同步、多用户与远程访问能力
- v1 视频剪藏以现有已支持的平台为范围,其他平台为预留扩展
- 归档目录命名以"时间戳_标题"为基本规则,冲突时自动追加数字后缀;标题清洗规则由实现层约定
- 索引文件以项目现有格式为准,本规格不重新定义其结构
- 移动端与图形界面不在本规格范围内,用户交互以命令行为唯一入口
